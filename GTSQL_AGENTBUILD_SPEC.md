# GTSQL Agent-Build Spec

A buildable specification for an autonomous coding agent to implement the
`TABULATE` family of clauses (defined in [GTSQL_PLAN.md](GTSQL_PLAN.md)) inside
the Rust [posit-dev/ggsql](https://github.com/posit-dev/ggsql) codebase, with
**rendered output that is byte-for-byte equivalent to `gt::gt()`'s default HTML
rendering** for the cases enumerated in [GTSQL_EXAMPLES.qmd](GTSQL_EXAMPLES.qmd).

This document is the agent's contract. It defines: the goal, the oracle, the
test loop, the sandbox, the stop conditions, and the phase order. Do not
deviate from it without recording the deviation in `AGENT_LOG.md`.

---

## 1. North Star

> For every example pair `(gt R code, GTSQL query)` in [GTSQL_EXAMPLES.qmd](GTSQL_EXAMPLES.qmd),
> running the GTSQL query through the Rust implementation must produce HTML
> that matches the HTML produced by the gt R code, modulo a documented set of
> normalizations (whitespace, id-attribute suffixes, etc.).

"gt is the spec" — not a paraphrase of gt, not a re-design. If gt renders
`$1,250.00` as `&#36;1,250.00` inside a `<td class="gt_row gt_right">`, GTSQL's
Rust writer must produce the same. The HTML produced by the upstream `gt`
package is the **ground truth oracle**.

### Non-goals

- Multi-output writers (LaTeX, RTF, Word, Markdown). HTML only for Phase 1–3.
- Features explicitly listed as omissions at the bottom of
  [GTSQL_EXAMPLES.qmd](GTSQL_EXAMPLES.qmd).
- Inventing new clauses or relaxing the syntax in [GTSQL_PLAN.md](GTSQL_PLAN.md).
- Performance work beyond "renders a 1k-row table in under a second."

---

## 2. The Oracle Pipeline (how "match gt exactly" is operationalized)

The hard part of "make it look like gt" is that gt's HTML is large, opaque,
and full of generated class names. You cannot eyeball-compare it. The agent
needs an automated diff with a precise normalization rule.

### 2.1 Fixture format

Each example becomes a triple under `tests/fixtures/<name>/`:

```
tests/fixtures/<NN_short_name>/
  query.ggsql            # the GTSQL query (copied verbatim from the .qmd)
  data.parquet           # the input rows (deterministic, no random seeds)
  expected.html          # gt's rendered HTML, captured once, checked in
  meta.toml              # source dataset, gt version, capture timestamp,
                         # normalization profile to use
```

`expected.html` is produced **once**, by an R script run by a human, captured
from real `gt` invocations. It is not regenerated by the agent. If gt changes
its output between versions, that is a deliberate, reviewed update — the
agent does **not** touch `expected.html`.

### 2.2 The fixture generator (one-time, human-run)

A standalone R script `scripts/capture_fixtures.R` walks
[GTSQL_EXAMPLES.qmd](GTSQL_EXAMPLES.qmd), executes the **gt** code block for
each example, materializes the underlying data into Parquet, and writes the
`gt::as_raw_html()` output to `expected.html`. This script lives outside the
agent loop. It is the one place where R is required.

Provided to the agent as a pre-built artifact. The agent **must not modify
captured fixtures.** It modifies Rust code until the comparison passes.

### 2.3 Normalization (the diff rule)

`gt` embeds randomly-suffixed CSS class names like `gt_table_abc123`. A
direct string compare will fail trivially. Define a Rust normalizer that, on
both sides of the diff, applies in order:

1. Strip the `id="abc..."` attribute on the root `<table>`.
2. Replace any class name matching `gt_table_[a-z0-9]+` with `gt_table_X`.
3. Collapse runs of whitespace inside tag attributes to a single space.
4. Collapse runs of whitespace between tags to a single newline.
5. Sort attributes alphabetically within each tag.
6. Re-escape entities to a canonical form (`&` → `&amp;`, `<` → `&lt;`, etc.).

This normalizer lives in `ggsql/src/tabulate/test_normalize.rs`. Both the
expected and actual HTML pass through it before comparison. Diffs are reported
as line-based unified diffs.

If a real semantic difference exists (e.g., a date format string in a locale
gt handles differently from `chrono`), it goes in the per-fixture `meta.toml`
as an `allowed_diff = ["..."]` regex, **with a justification comment**. The
agent may add such entries but each one must be referenced in `AGENT_LOG.md`.

## 3. Where the code goes in ggsql

Read [posit-dev/ggsql/CLAUDE.md](https://github.com/posit-dev/ggsql/blob/main/CLAUDE.md)
first. The relevant subtrees:

| Subtree | What lives there | Agent's job |
|---|---|---|
| `tree-sitter-ggsql/grammar.js` | tree-sitter grammar for the language | Add `TABULATE`, `FORMAT`, `FACET`, `SCALE` (re-use existing node), `HIGHLIGHT`, `LABEL` (extend existing) production rules. Add `corpus/tabulate/*.txt` parse tests. |
| `src/ast/` | typed AST nodes | Add `TabulateStmt`, `FormatClause`, `FacetClause`, `HighlightClause`, plus enums for `SPAN`/`STUB`, settings keys, palette names. |
| `src/parser/` | tree-sitter → AST lowering | Add visitors for the new node types. |
| `src/semantics/` | name resolution, column type checking, spanner-id namespace | Validate that `FORMAT STUB`/`SPAN` references resolve, that `SCALE FROM (a,b)` numerics are compatible with target column types, etc. |
| `src/tabulate/` | **new** — the table model | Build a typed in-memory `Table` IR (columns, spanners, stub, row groups, summary rows, cell styles, formatters) that is the Rust analogue of a `gt_tbl` object. |
| `src/tabulate/format.rs` | **new** — `{:num ...}` / `{:time ...}` / `{:Title}` etc. | Implement the format mini-language. Match `gt::fmt_*()` defaults (e.g. `fmt_currency(currency = "USD", decimals = 2)` ≡ `'${:num ,.2f}'`). |
| `src/tabulate/html.rs` | **new** — HTML writer | Produce HTML matching `gt::as_raw_html()` for the supported feature set. Re-use gt's CSS class names verbatim. |
| `src/tabulate/palettes.rs` | **new** — color palette catalogue | Reuse the existing VISUALISE palette table; expose by name. |
| `tests/tabulate/` | **new** — integration tests | One test per fixture under `tests/fixtures/`. Each test parses, executes against DuckDB, renders, normalizes, diffs. |

### gt-compatible CSS

`gt` ships a single embedded `<style>` block. The agent must vendor a copy at
`src/tabulate/gt_default.css` extracted from the gt R package (one-time, via
`scripts/extract_gt_css.R`). The Rust writer emits this CSS verbatim at the
top of every table. **Do not rewrite or "improve" the CSS.** Treat it as
opaque bytes.

## 4. The Agent Iteration Loop

The agent runs this loop until `make check` is green and all fixtures pass:

```
loop:
  1. Pick the lowest-numbered failing fixture in tests/fixtures/.
  2. Read its query.ggsql, expected.html, meta.toml.
  3. Identify the clauses involved (TABULATE / FORMAT / FACET / SCALE / HIGHLIGHT / LABEL).
  4. Make the minimum code change in src/ to advance that fixture's diff.
  5. Run: cargo build && tree-sitter generate && tree-sitter test
            && cargo test --test tabulate_fixtures -- <fixture>
  6. If the targeted fixture passes:
       run the full suite: make check
       if green → commit on the feature branch with a one-line message
       if a previously-green fixture regressed → revert step 4, try a smaller change
  7. If after 5 attempts the targeted fixture has not advanced (diff is the
     same or larger), write a 3-line note to AGENT_LOG.md describing the
     blocker and move to the next fixture. Surface unresolved blockers to the
     human at the end of the run.
```

### `make check` (the canonical pass gate)

```makefile
check: fmt-check clippy ts-test cargo-test fixtures

fmt-check:
	cargo fmt --check

clippy:
	cargo clippy --all-targets -- -D warnings

ts-test:
	cd tree-sitter-ggsql && tree-sitter generate && tree-sitter test

cargo-test:
	cargo test --workspace --no-fail-fast

fixtures:
	cargo test --test tabulate_fixtures -- --include-ignored
```

The agent terminates **successfully** only when `make check` returns 0.

### Per-iteration budget

- Max 50 iterations per feature branch before stopping for human review.
- Max 10 minutes of wall-clock per iteration (kill `cargo` and try a smaller
  change if exceeded).
- Hard stop after 6 wall-clock hours.

## 5. Phase ordering (mandatory)

The agent **must** complete each phase before starting the next. Each phase
is gated by a subset of fixtures from [GTSQL_EXAMPLES.qmd](GTSQL_EXAMPLES.qmd).

| Phase | Fixtures (by example number) | Capability proved |
|---|---|---|
| 0 | (none — setup) | Grammar skeleton, AST stubs, fixture harness, CSS vendored, normalizer working on a hand-written "hello world" table. |
| 1 | 1, 2, 9 | `TABULATE *`, column selection, `FORMAT col SETTING hide => true`. Default cell rendering with no formatters. |
| 2 | 3, 4, 5 | `FORMAT STUB`, `LABEL title/subtitle/caption`, basic `LABEL <col> =>` column relabel. |
| 3 | 6, 7, 8 | `FORMAT SPAN ... AS <id>` including nested spanners (example 8). |
| 4 | 10 | `FORMAT ... SETTING width/align`. |
| 5 | 11–17, 21 | Full `RENAMING * => '{:num ...}'` and `'{:time ...}'` formatter mini-language; per-column locale. |
| 6 | 18, 19, 20 | `RENAMING null =>`, `RENAMING 0 =>`, literal value `RENAMING 'Y' =>`. |
| 7 | 22, 23, 24, 25 | `SCALE background FROM ... TO ... [VIA ...]` including auto-domain, named palettes, transforms. |
| 8 | 26, 27, 28 | `HIGHLIGHT` with `FILTER` predicates over single and multiple columns; multiple HIGHLIGHTs per query. |
| 9 | 29, 30 | `FACET` row groups + summary rows with aggregate functions and labels. |
| 10 | 31, 32, 33 | `'{:Title}'`, `SETTING units`, forced-sign numeric formatters. |
| 11 | 34 | Comprehensive integration: header + spanner + formatting + scale + highlight + summary + facet in one query. |

A phase is "done" when every fixture in it passes `make fixtures` under
strict normalization (no new `allowed_diff` entries that aren't justified in
`AGENT_LOG.md`).

## 6. Sandbox (where the agent runs)

The agent operates inside a devcontainer in a fork of `posit-dev/ggsql`, not
inside this repo. This repo is the **read-only spec source**; the fork is the
workspace.

### Devcontainer requirements

`.devcontainer/devcontainer.json` in the ggsql fork:

- Base: `mcr.microsoft.com/devcontainers/rust:1`
- Features: Node 20 (for tree-sitter CLI), Python 3.11 (for any helper
  scripts; **not** an implementation language).
- Mounts: the fork; a read-only mount of this `gtsql` repo at `/spec`.
- Post-create: `cargo install tree-sitter-cli --locked && rustup component add rustfmt clippy && make -C /spec sync-fixtures`.

### Network policy

- Allowed outbound: `crates.io`, `static.crates.io`, `registry.npmjs.org`,
  `github.com` (read-only for `git fetch`).
- Blocked: everything else. No data exfiltration, no calls to remote LLMs from
  inside the build, no random package installs.

### Filesystem policy

- Writable: the fork's working tree, `target/`, `~/.cargo/registry`.
- Read-only: `/spec` (this repo).
- The agent never runs `git push origin main`, never force-pushes, never
  rewrites history. It pushes only to feature branches named
  `agent/tabulate-phase-<N>`. Humans merge PRs.

### Runner

Run inside the devcontainer with one of:

- VS Code Copilot agent mode (interactive).
- `codex` CLI in `--sandbox workspace-write` (the OS-level sandbox that
  mcp-repl's `--sandbox inherit` was designed for).
- Claude Code with the project's `CLAUDE.md` extended to point at this spec.

**mcp-repl is not in the loop for Rust work.** It is the wrong tool for
edit-build-test cycles. It stays in *this* repo for occasional fixture
inspection (load `expected.html` into BeautifulSoup, diff manually, etc.)
when a human wants to investigate a stubborn diff. That is a human aid, not
part of the autonomous loop.

## 7. Inputs handed to the agent on each run

1. This file (`GTSQL_AGENTBUILD_SPEC.md`).
2. [GTSQL_PLAN.md](GTSQL_PLAN.md) — the language reference.
3. [GTSQL_EXAMPLES.qmd](GTSQL_EXAMPLES.qmd) — the example corpus.
4. The pre-captured `tests/fixtures/**` tree (expected HTML + data).
5. The vendored `src/tabulate/gt_default.css`.
6. The current state of the ggsql fork.

System prompt for the agent (paste into Copilot/Codex/Claude Code config):

```
You are implementing TABULATE in posit-dev/ggsql per
/spec/GTSQL_AGENTBUILD_SPEC.md. Your ground truth is the captured HTML in
tests/fixtures/*/expected.html. Your job is to make `make check` pass.

Hard rules:
- Never modify tests/fixtures/*/expected.html.
- Never modify src/tabulate/gt_default.css.
- Never weaken the normalizer to make a diff pass.
- Never add an `allowed_diff` entry without writing a justification line in
  AGENT_LOG.md.
- Commit only to a feature branch. Open a PR when a phase is green.
- Follow the phase order in section 5 strictly.

On each iteration: pick the lowest-numbered failing fixture, change the
minimum code needed, re-run the targeted test, then `make check`. If a
previously-green fixture regresses, revert and try a smaller change.
```

---

## 8. Success criteria

The implementation is "done" when **all** of:

1. Every fixture in `tests/fixtures/` passes under strict normalization, with
   any `allowed_diff` entries justified in `AGENT_LOG.md` and ≤ 5 entries
   total across all fixtures.
2. `make check` returns 0 on a fresh devcontainer with no cache.
3. The integration example (fixture 34) renders correctly end-to-end.
4. A human (you) has reviewed and merged the per-phase PRs.

---

## 9. What this spec deliberately does **not** do

- Does not specify Rust struct layouts, trait hierarchies, or module
  internals. The agent decides those, constrained only by "match gt's HTML."
- Does not duplicate [GTSQL_PLAN.md](GTSQL_PLAN.md). When in doubt about
  semantics, the plan is authoritative.
- Does not specify how to handle features outside the example corpus. If a
  user query uses an unsupported feature, the parser should produce a clean
  "not yet implemented" error pointing at the relevant clause.

## 10. Bootstrapping checklist (one-time human work before the agent runs)

- [ ] Fork `posit-dev/ggsql` to your GitHub account.
- [ ] Clone the fork as a sibling folder to this repo.
- [ ] Add `.devcontainer/` per section 6.
- [ ] Write `scripts/capture_fixtures.R` and run it to generate `tests/fixtures/`.
- [ ] Write `scripts/extract_gt_css.R` and run it to produce `src/tabulate/gt_default.css`.
- [ ] Write the `make check` target and the fixture-diff test harness skeleton
      (one passing hello-world fixture proves the harness works).
- [ ] Commit the bootstrap on `main` of the fork.
- [ ] Create branch `agent/tabulate-phase-0` and hand control to the agent.

Once the bootstrap is in, every subsequent phase is the agent's job.
